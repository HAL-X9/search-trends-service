# Search trends service

Высокопроизводительный event-driven микросервис на Go, предназначенный для агрегации, фильтрации и расчета аналитических
метрик поисковых запросов в реальном времени с целью формирования виджета "Сейчас ищут".

Архитектура сервиса спроектирована с учетом требований к системам с высокой интенсивностью запросов (Highload) и ориентирована 
на минимизацию задержек (Low Latency) при асимметричной нагрузке, где чтение (Read Intensive) превалирует над записью в 10–50 раз.

## Быстрый старт

Этот раздел поможет запустить проект локально, без ручной настройки Kafka и топиков.
Все необходимые сервисы поднимаются автоматически, через Docker Compose.

### Системные требования
- Docker Desktop
- Открытые порты: 29092 (для kafka), 8080 (для HTTP API сервиса)

1.  **Клонируйте репозиторий и перейдите в корень проекта:**
    ```bash
    git clone [https://github.com/HAL-X9/search-trends-service.git](https://github.com/HAL-X9/search-trends-service.git)
    cd search-trends-service
    ```

2.  **Запустите сборку и поднимите весь стек одной командой:**
    ```bash
    docker-compose up --build
    ```

3. **Проверка корректности работы (Что искать в логах):**
    Скрипт инициализации `kafka-init-topics` заблокирует старт генератора и основного сервиса до тех пор, пока топик гарантированно не создастся на брокере. После успешного старта в консоли побегут параллельные логи:
    * `traffic-generator-1` — сообщает об успешной отправке случайных фраз раз в секунду.
    * `search-trends-service-1` — логирует получение данных и триггерит антифрод-фильтрацию (блокировку бан-слов из стоп-листа).
    
## Контракты данных

Для интеграции со смежной командой поиска сформирован и зафиксирован следующий контракт сообщения (JSON):

```json
{
  "query": "белые кроссовки nike",
  "user_id": "usr_948201asdf",
  "ip_address": "192.168.1.50",
  "timestamp": 1779543595
}
```

Обоснование полей контракта:
- query (string) Исходный поисковой запрос (нормализуется на стороне сервиса)
- user_id и ip_address (string) Поля необходимы для защиты от ботов, накруток, парсеров.
Позволяют индифицировать сессию уникального пользователя или бота. Если один идентификатор генерирует аномальный всплеск одинаковых запросов, 
система дедуплицирует их в рамках окна, предотвращая искажение трендов.
- timestamp (int64) Unix-время совершения поискового действия пользователем. Необходим для детерминированной работы алгоритма скользящего окна.
