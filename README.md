# Search trends service

Высокопроизводительный event-driven микросервис на Go, предназначенный для агрегации, фильтрации и расчета аналитических
метрик поисковых запросов в реальном времени с целью формирования виджета "Сейчас ищут".

Архитектура сервиса спроектирована с учетом требований к системам с высокой интенсивностью запросов (Highload) и ориентирована 
на минимизацию задержек (Low Latency) при асимметричной нагрузке, где чтение (Read Intensive) превалирует над записью в 10–50 раз.

## Быстрый старт

Этот раздел поможет запустить проект локально, без ручной настройки Kafka и топиков.
Все необходимые сервисы поднимаются автоматически, через Docker Compose.

### Системные требования
- Docker Desktop
- Открытые порты: 29092 (для kafka), 8080 (для HTTP API сервиса)

1.  **Клонируйте репозиторий и перейдите в корень проекта:**
    ```bash
    git clone https://github.com/HAL-X9/search-trends-service.git
    cd search-trends-service
    ```

2.  **Запустите сборку и поднимите весь стек одной командой:**
    ```bash
    docker-compose up --build
    ```

3. #### **Проверка корректности работы (Что искать в логах):**
    Скрипт инициализации `kafka-init-topics` заблокирует старт генератора и основного сервиса до тех пор, пока топик гарантированно не создастся на брокере. После успешного старта в консоли побегут параллельные логи:
    - `traffic-generator-1` сообщает об успешной отправке случайных фраз раз в секунду.
    - `search-trends-service-1` логирует получение данных и триггерит антифрод-фильтрацию (блокировку бан-слов из стоп-листа).  

    #### Проверка Топа запросов

    `curl -i "http://localhost:8080/api/v1/trends/top?limit=10"`

    #### Добавить слово в stop-list

    ```
    curl -i -X POST "http://localhost:8080/api/v1/stop-list" \
     -H "Content-Type: application/json" \
     -d '{"word":"спам спам спам"}'
   ```

   #### Проверить, что слово сохранено

    Локально:
    `cat ./config/stop-list.txt`

    Или внутри контейнера:
    `docker compose exec search-trends-service sh -lc 'cat config/stop-list.txt'`

    #### Удалить слово из stop-list

    ```
    curl -i -X DELETE "http://localhost:8080/api/v1/stop-list" \
      -H "Content-Type: application/json" \
      -d '{"word":"спам спам спам"}'
   ```
    
## Контракты данных

Для интеграции со смежной командой поиска сформирован и зафиксирован следующий контракт сообщения (JSON):

```json
{
  "query": "белые кроссовки nike",
  "user_id": "usr_948201asdf",
  "ip_address": "192.168.1.50",
  "timestamp": 1779543595
}
```

Обоснование полей контракта:
- query (string) Исходный поисковой запрос (нормализуется на стороне сервиса)
- user_id и ip_address (string) Поля необходимы для защиты от ботов, накруток, парсеров.
Позволяют идентифицировать сессию уникального пользователя или бота. Если один идентификатор генерирует аномальный всплеск одинаковых запросов, 
система дедуплицирует их в рамках окна, предотвращая искажение трендов.
- timestamp (int64) Unix-время совершения поискового действия пользователем. Необходим для детерминированной работы алгоритма скользящего окна.

## Архитектура и технические решения

Сервис построен по event-drive модели, и в первую очередь нацелен на обеспечение максимлаьной производительности.
В проекте реализуется классическая слоистая архитектура (handler - usecases - repository). Данное разделение довольно простое для чтения, масштабирования
и поддержки.

Для агрегации запросов был выбран алгоритм скользящих окон с кольцевым буфером. На каждую секунду выделяется выделяется бакет и при каждом запросе счетчик инкрементируется.
Раз в секунду окно сдвигается, самый старый бакет очищается. Данный подход имеет фиксированую память, что позволяет не создавать лишних аллокаций и следовательно не вызывать лишний раз сборщик мусора,
обеспечивая высокую скорость обработки запроса.

> Подробнее про выбор алгоритма, библиотек и архитектуры, trade-off-ы можно прочитать в [ADR-0001](./docs/0001-service-architecture.md)

### Фильтрация и защита от накруток

Для Фильтрации бан-слов была выбрана архитектура персистентного хранилища в оперативной памяти, на базе плоского .txt файла.

**Почему не Redis или любое другое хранилище?**

При проектировании системы акцент был сделан в первую очередь на производительность и минимальный latency.
На старте приложения, стоп-лист вычитывается из локального файла и загружается прямо в память приложения `map[string]struct{}`.
Проверка входящих запросов из Kafka происходит в оперативной памяти под защитой мьютекса за константное время O(1), что полностью исключает сетевые задержки.

При аварийном или плановом рестарте сервиса актуальное состояние стоп-листа полностью восстанавливается в RAM за миллисекунды при холодной загрузке.

Redis в свою очередь имеет сетевые задержки так как работает на отдельном процессе, к тому же придется продумывать
механизмы персистентности, репликации и восстановления данных, например RDB snapshots, AOF, сохранение реплик в бд или гибридный подход, что сильно увеличит инфраструктурную сложность.

**Trade-off**
Компромиссами за экстремальную производительность и инфраструктурную простоту стали следующие ограничения:
- Горизонтальное масштабирование: Текущий подход жестко завязан на single-instance архитектуру. 
Если запустить несколько реплик сервиса, то добавление слова в стоп-лист через HTTP-API обновит файл и память только на одной конкретной ноде. 
Остальные реплики об этом не узнают.
- Оперативная память: Так как весь стоп-лист держится в памяти, размер словаря ограничен доступной RAM.
В рамках задачи (бан-слова) это не проблема, так как даже миллион слов займет всего несколько десятков мегабайт, но подход не масштабируется на терабайты данных.
- Консистентность диска при удалении: Для сохранения высокой скорости записи (O_APPEND), операция удаления слова из стоп-листа происходит только в оперативной памяти (в файле запись физически остается). 
Полная синхронизация файла происходит только при плановом перезапуске сервиса.
