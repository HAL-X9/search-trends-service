# ADR 0002: Гибридное отказоустойчивое хранилище стоп-листа (Persistent In-Memory)

## Статус
Принято (Approved)

## Контекст
Сервис обработки поисковых трендов должен фильтровать нежелательный контент, бан-слова с минимальными задержками.
Согласно бизнес требованиям, администраторы/маркетологи должны иметь возможность динамически изменять стоп-лист через API
без перезапуска сервиса.

При проектировании возникают следующие ограничения и риски:
- Хранение стоп-листа исключительно в оперативной памяти процесса, хоть и обеспечивает высокую скорость O(1), однако приводит
  к полной потери данных при плановом или аварийному перезапуске приложения/пода/контейнера.
- Использование внешних СУБД, как Redis или PostgreSQL, решает проблему персистентности, но вносит оверхед, расходы на сериализацию
  и риск деградации пропускной способности.
- Согласно ТЗ, использование managed-решений ограничено, а стек должен быть оптимизирован под Highload.

## Решение
Решено реализовать персистентное in-memory хранилище(Persistent In-Memory Storage) с синхронным снапшотом, на базе
локального файла (`config/stop-list.txt`).

Архитектура решения состоит из двух слоев:
1. Runtime Layer (In-Memory):тПотокобезопасная структура `map[string]struct{}` под управлением `sync.RWMutex` (`atomic.Value` при редких мутациях). Все входящие поисковые логи валидируются только против этого слоя за константное время O(1).
2. Persistence Layer (Flat File): Локальный файл на диске, выступающий в роли единственного источника правды (Single Source of Truth) при холодной загрузке.

Mutation (Добавление/Удаление через API):
Когда поступает запрос на эндпоинт `POST /api/v1/admin/stop-list/words`, транзакция выполняется конкурентно:
1. Слово атомарно добавляется в In-Memory мапу, чтобы изменения вступили в силу для потока фильтрации мгновенно.
2. Слово дозаписывается в конец файла `stop-list.txt` в режиме `O_APPEND` с принудительным вызовом `fsync` (или асинхронно через буферизированный канал).

## Последствия (Trade-offs)

**Плюсы**:
- Максимальная производительность: Проверка миллионов поисковых логов не утилизирует сеть и не зависит от доступности сторонних баз данных.
- Отказоустойчивость: При перезапуске контейнера в Kubernetes/Docker состояние стоп-листа восстанавливается до актуального за миллисекунды.
- Простота инфраструктуры: Нет необходимости разворачивать, мониторить и бэкапить инстанс Redis.

**Минусы**: 
- Горизонтальное масштабирование: Если мы запустим N реплик нашего сервиса, у каждой будет свой локальный файл на диске. 
Запрос POST на одну реплику изменит стоп-лист только на ней.
- Так как в рамках данного ТЗ сервис деплоится в единственном экземпляре, проблема распределенного стейта опускается. 
- При переходе на кластерную архитектуру данный файл будет заменен на чтение из распределенной конфигурации (например, Etcd / Consul) 
или через трансляцию событий изменения стоп-листа в отдельный топик той же Kafka.
